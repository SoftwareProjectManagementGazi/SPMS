# Fake-Pass Test Audit — SPMS

> **What this is.** An exhaustive audit of every backend and frontend test in the repository, hunting for *fake passes* — tests that pass but would **keep passing even if the production code they claim to cover were broken**. Such tests give false confidence: green CI that verifies nothing.

- **Date:** 2026-05-29
- **Generated by:** the `fake-pass-test-audit` multi-agent workflow (`.claude/workflows/fake-pass-test-audit.js`)
- **Scale:** 550 agents · ~19.6M tokens · ~62 min · **249 test files, 100% covered** (0 dropped, 0 source-location failures)
- **Method:** one agent per test file read the test **and its production source**, then asked *“if I broke the source, would this test fail?”* Every finding had to name a concrete **mutation it would miss** (a one-line source bug the test stays green against). Every medium-or-worse finding was then **independently re-verified by a separate Opus agent instructed to refute it** — 36 findings were rejected as false positives.

## The #1 systemic issue: `xfail` + `assert False` stub tests

By far the largest and most dangerous class of fake pass. The pattern:

```python
@pytest.mark.xfail(strict=False, reason="not implemented yet")
async def test_delete_comment_by_non_author_raises_forbidden():
    assert False, "not implemented"
```

**Why it is a fake pass, not an honest TODO:**

1. `assert False` makes the test fail; `xfail` flips that failure into **XFAIL**, which pytest reports as a *passing* run by default. The test contributes to your green bar.
2. Because `strict=False`, even if the feature later works and the body starts passing, the result is **XPASS — still not a failure**. So the marker never forces anyone to finish the test.
3. Critically, **the production code these stubs name is already implemented and live.** The audit confirmed, per stub, a one-line source mutation (e.g. *remove the author-id authorization check*, *empty the blocked-extensions set*, *skip the recurring-instance creation*) that ships to users **with zero test signal**. These are not “tests for unbuilt features” — they are real, security-relevant behaviors left unguarded behind a green stub.

**Backend files with confirmed `disabled-but-counted` findings** (20 files) — `xfail`/`assert False` stubs plus a handful of other skipped/disabled tests. Of these, **11 files** carry the literal `xfail(… "not implemented" …) + assert False` stub (grep-confirmed); the rest hold other disabled tests counted as coverage:

Add a CI guard: `grep -rn 'assert False' Backend/tests && exit 1`, and forbid `xfail(strict=False)` (require `strict=True` so an unexpectedly-passing test fails loudly and gets finished).

## Recommendations (in priority order)

1. **Purge the `xfail`/`assert False` stubs (critical).** 11 files, ~45 stubs. Implement the real test or remove it. Add CI guards: fail the build on `assert False` in `tests/`, and ban `xfail(strict=False)`.
2. **Fix the RBAC permission-seeding infrastructure (high, security-relevant).** Multiple integration tests assert the 38-permission matrix but never seed it — they pass only on a pre-populated shared DB and would not catch a removed permission. Seed inside the test's own transactional session (or provision the test DB via `alembic upgrade head` instead of `Base.metadata.create_all`) and derive expected counts from `PERMISSIONS_SEED`, not the literal `38`.
3. **Strengthen status-code-only API tests (high/medium).** After a mutating request, assert the response body **and** the persisted side effect (re-GET, or query the DB), not just `status_code == 2xx`.
4. **Remove E2E escape hatches (high).** Replace `if (await x.isVisible()) { …asserts… }` and `.catch(() => {})` skip-guards with real waits + unconditional assertions; fix the selectors that silently never match (e.g. the priority control is a button-popover, not a `<select>`). Make the backend-health skip-guard probe the actual backend, or the test is decorative.
5. **Replace weak-only matchers (medium).** `toBeDefined`/`toBeTruthy`/`not.toThrow`/`is not None` on the *central* claim → assert the actual value/state the test name promises.
6. **Adopt mutation testing as the durable guard.** A fake pass is invisible to coverage tools (it executes the line, just doesn't verify it). Mutation testing is the real defense:
   - **Backend:** [`mutmut`](https://mutmut.readthedocs.io/) or `cosmic-ray` over `app/application` + `app/domain`.
   - **Frontend:** [StrykerJS](https://stryker-mutator.io/) over `lib/`, `hooks/`, `services/`.
   - Start with the hotspot files in the table above; a surviving mutant = a fake pass this static audit may have missed.
